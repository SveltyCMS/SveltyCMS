# Content System Architecture (2026 Edition)

The SveltyCMS Content System is a high-performance, reactive engine responsible for managing collection schemas and the content navigation tree. The 2026 refactor emphasizes **incremental updates**, **zero-restart development**, and **real-time synchronization** while maintaining strong separation of concerns.

## 🏗️ The 5 Pillars

### 1. Reactive Store (Client)

**File**: `src/stores/content-store.svelte.ts`  
Uses **Svelte 5 Runes** (`$state`, `$derived`) as the single source of truth for the content tree. Provides instant, granular UI updates when nodes change.

### 2. Public API Facade (Universal)

**File**: `src/content/index.ts`  
Exposes `contentSystem` (with `contentManager` as backward-compatible alias). Handles initialization, context (`useContent`), caching, and delegates to server services. Fully integrated with the **Local SDK** (`locals.cms`).

### 3. Incremental Reconciliation (Server)

**File**: `src/content/content-service.server.ts`  
The intelligent core. Uses **mtime-based scanning** + **schema diffing** (`schemasAreEqual`) to skip unchanged files and minimize database writes (up to 90% reduction in I/O).

### 4. Smart Module Processing (Server)

**File**: `src/content/module-processor.server.ts` (or inline in service)  
Safe sandboxed parsing of compiled collection modules with widget proxy support. Only executed when mtime changes.

### 5. Real-time Synchronization

**Files**:

- `src/content/content-watcher.server.ts` (dev-only, chokidar)
- `src/content/content-sse.svelte.ts` (client-side SSE)

In development: file changes → instant incremental reload.  
In production: Server-Sent Events push updates to connected clients, replacing polling.

## 🚀 Performance Optimizations

1.  **mtime Scanning**: Only reads and parses files that have a newer modification timestamp than the cache.
2.  **Schema Diffing**: Compares the scanned schema with the database version; if no fields or metadata changed, the DB update is skipped entirely.
3.  **SSE Push**: Eliminates 5-second polling intervals, reducing client-side CPU usage and server-side log noise.
4.  **Local SDK Injection**: Internal CMS calls bypass the network stack, resulting in **< 0.15ms latency** for core operations.

---

## 🛠️ Developer Experience

- **Zero Restart**: Adding a new collection file in `config/collections` triggers the watcher, which reconciles the DB and updates the UI instantly.
- **Strong Typing**: The `LocalCMS` SDK provides full TypeScript types for all collection operations, from creation to complex joins.
- **Tenant Isolation**: All content operations automatically respect the `tenantId` context provided by the middleware.
