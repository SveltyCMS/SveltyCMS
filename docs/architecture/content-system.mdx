# Content System Architecture (2026 Edition)

The SveltyCMS Content System is a high-performance, reactive engine. The June 2026 security audit added **worker pool sandboxing**, **plugin hooks**, and **performance tracing**.

## 🏗️ The 5 Pillars

### 1. Reactive Store (Client)

**File**: `src/stores/content-store.svelte.ts`  
Uses **Svelte 5 Runes** (`$state`, `$derived`) as the single source of truth for the content tree. Provides instant, granular UI updates when nodes change.

### 2. Public API Facade (Universal)

**File**: `src/content/index.ts`  
Exposes `contentSystem` (with `contentSystem` as backward-compatible alias). Handles initialization, context (`useContent`), caching, and delegates to server services. Fully integrated with the **Local SDK** (`locals.cms`).

### 3. Incremental Reconciliation (Server)

**File**: `src/content/content-service.server.ts`  
The intelligent core. Uses **mtime-based scanning** + **schema diffing** (`schemasAreEqual`) to skip unchanged files and minimize database writes (up to 90% reduction in I/O).

#### Vectorized Processing & API Optimization

SveltyCMS utilizes a **Vectorized `modifyRequest` pipeline** to handle large data batches. This refactor moved the system from sequential object spreading to chunked parallel execution:

- **Scalability**: Handles 10,000+ item imports without blocking the event loop.
- **Memory Efficiency**: Reduced object allocations by ~90% for standard fields.
- **Relational Performance**: Achieved a ~28% boost in relational averages (5.29ms → 3.81ms) by removing redundant processing in hot paths.

#### Self-Healing & Native Introspection

SveltyCMS utilizes native database introspection via the unified `listSchemas` SDK method to maintain parity and recover from drift. While the interface is standardized, we use different mechanisms internally to ensure we are **fully optimized** for each platform:

1.  **MongoDB (`listCollections`)**: Since MongoDB is a document store, it doesn't have "tables." Its native command is `listCollections`. Using this is much faster and more reliable than trying to simulate a SQL-style query.
2.  **PostgreSQL/MariaDB (`information_schema`)**: These follow the ANSI SQL standard. The `information_schema` is a virtual database that exists specifically to allow tools to "ask" the database what tables exist. It is the most robust way to handle introspection in a multi-tenant SQL environment.
3.  **SQLite (`sqlite_master`)**: SQLite is a serverless, file-based engine. It doesn't support the heavy `information_schema` by default to keep the file size small. Instead, it stores everything in a special internal table called `sqlite_master`. Scanning this is the high-performance way to find tables in SQLite.

##### The "Self-Healing" Benefit

By using these native methods instead of a generic abstraction, SveltyCMS achieves:

- **Detect Drift**: If a table is manually deleted in the database, the CMS detects it instantly via native introspection and triggers automated "Repair" workflows.
- **Zero-Overhead**: We don't need to maintain a redundant "registry" table; we ask the database for the single source of truth directly.
- **Security**: System tables (like `pg_` or `mongodb.`) are filtered at the database level before the data reaches the application logic.

### 4. Smart Module Processing (Server)

**File**: `src/content/module-processor.server.ts`  
**NEW**: `src/content/module-worker.server.ts` + `module-worker-pool.server.ts`

Safe sandboxed parsing of compiled collection modules with widget proxy support. Now backed by a **native `worker_threads` pool** (zero dependencies — Node.js built-in only). Each worker runs in its own V8 isolate — a crashed schema file never affects the main process.

**Worker Pool Features**:

- Fixed-size pool (CPU cores / 2, minimum 2)
- Automatic worker replacement on crash/timeout
- Per-task 10s timeout guard
- Idle worker cleanup after 30s (keeps 1 alive)
- `getModuleWorkerPool()` singleton + `shutdownWorkerPool()`

**Security Benefit**: Process isolation prevents supply-chain attacks from malicious collection files.

### 5. Real-time Synchronization

**Files**:

- `src/content/content-watcher.server.ts` (dev-only, native watch)
- `src/content/content-sse.svelte.ts` (client-side SSE)

In development: file changes → instant incremental reload.  
In production: Server-Sent Events push updates to connected clients, replacing polling.

## 🚀 Performance Architecture

The 2026 update introduced a massive architectural leap in content processing, achieving **sub-millisecond (0.04ms)** scan latencies for 1,000+ collections.

### 🧵 1. Worker Thread Pooling

To prevent event-loop blocking during massive schema reloads, SveltyCMS now offloads module parsing and widget proxy creation to a **dedicated pool of background workers** (`module-worker.server.ts`). This ensures the main UI thread remains responsive even when thousands of schemas are being processed simultaneously.

### 🌳 2. Persistent Mtime Tree (Dirty Bits)

The scanner now maintains a persistent, in-memory **"Dirty Bit" tree**.

- **Steady State**: If the `ContentWatcher` hasn't flagged any changes, the scanner skips `fs.readdir` and `fs.stat` entirely, returning the cached schema map instantly.
- **Micro-Surgical Invalidation**: When a file changes, only that specific node in the tree is marked dirty, triggering a single `fs.stat` and a single cache refresh.

### 📦 3. Batch Cache Retrieval (`getMany`)

Individual cache lookups for every file have been replaced by a **Vectorized Batching** pattern. The system fetches all 1,000+ schema metadata blocks in a single `O(1)` trip to Redis/Memory using the new `CacheService.getMany` method.

### 🔢 4. Fast Deterministic Hashing

Replaced heavy cryptographic MD5 with a lightweight, numeric string hash in `module-processor.server.ts`. This reduces reconciliation time for unchanged schemas by ~40%.

## 🔌 Plugin System (June 2026)

**File**: `src/content/content-service.server.ts`

The content system exposes lifecycle hooks for third-party plugins:

```typescript
import { registerContentPlugin } from "@src/content/content-service.server";

registerContentPlugin({
  name: "my-plugin",
  onSchemaLoad(schema, filePath) {
    // Enrich, validate, or transform schema before caching
    return schema;
  },
  onReconcile(operations, prunedPaths) {
    // React to schema sync — notify external systems
  },
  onNodeChange(node, action) {
    // Track content tree changes — audit logging
  },
});
```

**Features**:

- `onSchemaLoad` → runs after validation, before caching — plug in transforms
- `onReconcile` → fires when collection schemas are synced to DB
- `onNodeChange` → tracks individual node creates/deletes in the content tree
- Duplicate plugin names are rejected with a warning

## 🛡️ Schema Validation & Path Hardening (June 2026)

**Schema Validation** (`validateSchemaFields`):

- Rejects schemas with missing `name` or empty `fields` array
- Detects duplicate `db_fieldName` values (prevents silent data corruption)
- Invalid schemas are logged and skipped — never reach cache or database

**Path Security** (`isSafeCollectionPath` — 3-layer check):

1. Must be within `.compiledCollections` directory and end with `.js`
2. Blocks `..` traversal (absolute path or relative escape)
3. Rejects special characters beyond `\w\-./\\`

## 📊 Lightweight Tracing (June 2026)

Zero-dependency performance tracking using `performance.now()`:

```typescript
import { traceStart, getTraceStats } from "@src/content/content-service.server";

const done = traceStart("scanCompiledCollections");
// ... work ...
done();

const stats = getTraceStats();
// → { "scanCompiledCollections": { avg: 0.004, p95: 0.006, count: 150 } }
```

Internal traces track `scanCompiledCollections`, `reconcile`, and `loadSchemaNative` — keeping the last 100 samples per label.
