# Content System Architecture

The Content System is the core of SveltyCMS, responsible for managing, indexing, and serving content and collections. It leverages Svelte 5 runes for fine-grained reactivity and supports multi-tenant isolation at the filesystem level.

## Core Components

### 1. Content Manager (`src/content/content-manager.ts`)

The `ContentManager` is a singleton class that orchestrates the entire content lifecycle. It serves as the central point of access for:

- **Initialization**: Bootstrapping the content management system on server start.
- **Multi-Tenant Isolation**: Manages collection structures per tenant using `.compiledCollections/{tenantId}/`.
- **Reactivity (Svelte 5)**: Uses `$state()` to track `contentVersion` and navigation structures.
- **Access Control**: Dynamic filtering of collections based on user permissions and roles.

**Key Features:**

- **Singleton Pattern**: Ensures a single source of truth for content state across the server.
- **Svelte 5 Runes**: Uses `$state()` for deep reactivity and `$derived()` for computed navigation trees.
- **Multi-Tenant Folders**: Compiled collection JS files are stored in `.compiledCollections/{tenantId}/` to ensure strict isolation.
- **Progressive Loading**: Navigation nodes are fetched incrementally to support massive sites (>10k categories).
- **Diagnostics**: Built-in health checks (`getHealthStatus`) monitor cache performance and system health.
- **Stable ID Generation**: Deterministic hashing prevents hydration mismatches between SSR and client.

### 2. Content Index (`src/content/index.ts`)

This module handles the low-level details of content loading and processing.

- **Dynamic Imports**: Uses `import.meta.glob` (dev) and filesystem scanning (prod) to load collections.
- **Batch Processing**: Processes collections in concurrent batches to maximize startup speed.
- **Polling**: Lightweight client-side polling checks for `contentVersion` updates.

### 3. Collection Scanner (`src/content/collectionScanner.ts`)

A utility for production that scans `.compiledCollections/` for tenant-specific definitions. This bypasses dynamic imports in the built application.

### 4. Migration Engine (`src/services/MigrationEngine.ts`)

A database-agnostic service that synchronizes Code Definitions with the Database Schema.

- **Drift Detection**: Identifies changes between TypeScript schemas and DB collections.
- **Safe Mode**: Prevents accidental data loss during schema migrations.

## Reactivity Model (Svelte 5)

SveltyCMS uses a **Signal-based Reactivity Model**:

1.  **State Tracking**: The `ContentManager` exposes reactive state via Svelte 5 runes.
2.  **Version Propagation**: When content is modified, `contentVersion` ($state) increments.
3.  **Fine-Grained Updates**: Components using `collections` or `contentStructure` re-render only for the specific pieces of data that changed.
4.  **Client Polling**: The client polls `/api/content/version` every 10s. If it detects a mismatch, it triggers a reactive update of the local state.

---

## Caching & Performance

- **Layered Caching**: Combines Redis (L1) and Database (L2) for sub-ms schema lookups.
- **Prefetching**: Automatically warms collection caches on system start.
- **Hydration**: SSR transfers the initial state to the client, preventing initial layout shifts.

---

## Future Improvements

- **WebSocket Real-time**: Moving from polling to push-based updates.
- **Granular Permissions**: Per-field access control within the `ContentManager`.
- **Edge-Compiled Schemas**: Pre-compiling schemas to WASM for even faster execution.
