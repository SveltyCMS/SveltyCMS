# System State & Health Architecture

SveltyCMS employs a robust **State Machine** to manage the application's lifecycle, ensure data integrity, and handle system failures gracefully. This system acts as a gatekeeper for all incoming requests.

## 🧠 Core Concepts

In 2026, SveltyCMS migrated its core state management from traditional Svelte Stores to **Svelte 5 Runes** to eliminate the performance overhead of synchronous state lookups.

### Reactive State with Svelte 5 Runes

The system state is now managed by a reactive class (`SystemStateContainer`) utilizing `$state` and `$derived` runes. This architecture provides several enterprise-tier benefits:

1.  **Zero-Tax Synchronous Access**: Unlike standard stores which require the expensive `get()` helper (iterating all subscribers), Svelte 5 runes allow direct, zero-overhead access to the current state via `system.current`.
2.  **Fine-Grained Reactivity**: UI components only re-render when the specific service they are watching (e.g., `databaseStatus`) changes, rather than on every system update.
3.  **Derived Intelligence**: The `overallState` is a `$derived` rune, ensuring it is always calculated on-demand with zero computation tax when idle.
4.  **Backward Compatibility**: The system maintains a legacy `systemStateStore` (Writable) that is automatically synchronized with the reactive class, ensuring that older Svelte 4/Kit components and hooks continue to function without modification.

The state is enforced by the `handleSystemState` server hook, which gates all incoming requests based on the reactive status.

### System States

The application can be in one of the following states:

| State              | Description                                                       | Request Handling                                    |
| :----------------- | :---------------------------------------------------------------- | :-------------------------------------------------- |
| **`IDLE`**         | Initial state. System is waiting for setup or initialization.     | **Blocked** (except Setup Wizard & Health Checks)   |
| **`INITIALIZING`** | Database connection and core services are starting up.            | **Blocked** (requests wait for completion)          |
| **`RECOVERY`**     | **[NEW]** Autonomous healing in progress after a service failure. | **Blocked** (503 Service Unavailable + Retry-After) |
| **`READY`**        | Normal operational state. All systems go.                         | **Allowed**                                         |
| **`DEGRADED`**     | System is operational but some non-critical services are failing. | **Allowed** (with warning logs)                     |
| **`FAILED`**       | Critical failure (e.g., DB down). System cannot function.         | **Blocked** (503 Service Unavailable)               |

## 🧠 Intelligent Phased Initialization

SveltyCMS 2026 utilizes a **Context-Aware Phased Boot** to minimize resource waste and maximize startup speed. Instead of a monolithic boot, the system targets specific "Minimum Viable Services" based on the request intent.

### Boot Phases

1.  **`SETUP` Phase**: Establishes basic database connectivity. Results in system state **`SETUP`**. Required for the Setup Wizard. No heavy widgets or background jobs are loaded.
2.  **`CORE` Phase**: Loads Authentication, Security, and Critical Settings. Results in system state **`READY`**. Required for the Login screen. Triggers non-critical background services (Cache Warming, Indexing).
3.  **`FULL` Phase**: Finalizes the boot by performing **Predictive Cache Pre-Warming**, Widget Reconciliation, and Background Optimizations. Results in system state **`WARMED`**. Required for the CMS Dashboard.

This **Adaptive Phased Boot** ensures that the system is live and serving traffic within milliseconds, while heavy resource-intensive tasks are deferred to background threads.

> [!TIP]
> **Logical vs. Technical Lazy Loading**: While **Phased Initialization** handles the _logical_ order of service startup, SveltyCMS uses the **[Lazy Holder](./database/index.mdx#lazy-holder-architecture)** technical pattern to ensure physical code isolation, preventing server-only modules from leaking into client bundles.

## 🛡️ Request Gating Middleware

The `handleSystemState` hook (`src/hooks/handle-system-state.ts`) is the primary gatekeeper, intercepting every request to ensure system operational readiness.

### Logic Flow (Phases)

1.  **Fast-Path Resolution**: Static assets and well-known paths (SSCP, health checks) are resolved immediately to maximize performance.
2.  **Initialization Phase**:
    - If the system is `IDLE` but configured, it triggers a backgrounded initialization flow via `waitForInitialization`.
    - **Optimization**: For bootstrap UI routes (Login, Setup), requests are non-blocking to allow the frontend to load while the backend warms up.
3.  **Bootstrap & Host Validation**:
    - Ensures that during restricted states, only authorized hosts (localhost, ORIGIN, or Demo) can access sensitive bootstrap routes.
    - Prevents unauthorized access to `/setup` or `/login` during initialization.
4.  **Restricted Access Handling**:
    - Pauses non-bootstrap requests if the system is `INITIALIZING` until the boot sequence completes (or times out at 60s).
    - Blocks traffic with state-specific 503 errors if the system is in `SETUP`, `MAINTENANCE`, or `FAILED` modes.
5.  **Final Readiness & Enrichment**:
    - Performs a final `isSystemReady()` check.
    - If `DEGRADED`, injects `event.locals.degradedServices` with a list of unhealthy dependencies for UI awareness.

## 🧠 Implemented Advanced Features

### 1. Autonomous Self-Recovery (The Watchdog)

SveltyCMS features an **Autonomous System Watchdog** (`src/services/system/watchdog.ts`) that acts as a background "guardian."

- **Heal-on-Failure**: If a critical service (Database or Auth) becomes `unhealthy`, the Watchdog automatically triggers a targeted re-initialization.
- **Exponential Backoff**: Recovery attempts use a progressive delay (5s, 10s, 20s) to prevent overloading infrastructure during transient network outages.
- **Drift Detection**: The Watchdog continuously compares the system's "internal map" of health against reality. If the system claims to be `READY` but the database connection is dead, it autonomously demotes the state to `RECOVERY` and initiates healing.

### 2. High-Resolution Boot Telemetry

Every boot phase is measured with `performance.now()` precision. These metrics are exposed via the health API and recorded in the `MetricsService`:

- `boot:phase:setup`: Latency for database handshake.
- `boot:phase:core`: Latency for auth/settings hydration.
- `boot:phase:full`: Latency for content/widget system reconciliation.
- `recoveryCount`: Number of times the system successfully healed itself without human intervention.

### 3. Robust Boot Protection (Self-Healing)

The system manages a one-shot initialization lifecycle using a `dbInitPromise`:

- **Timeout Safety**: Initialization has a hard 60-second limit. If services fail to start, the system enters the `FAILED` state to protect the underlying infrastructure.
- **Background Warming**: For high-concurrency environments, only the first request triggers the init flow; subsequent requests wait on the same shared promise, preventing "thundering herd" connection storms.
- **Cleanup on Failure**: Failed boot attempts log the full stack trace and inform the `metricsService` for alerting.

### 2. Adaptive Rate Limiting (Hardware-Aware)

SveltyCMS 2026 features a **Dynamic Hardware-Aware Rate Limiter** that adjusts request costs based on real-time system pressure.

- **System Monitoring**: The `SystemMonitor` utility (`src/utils/system-monitor.ts`) tracks CPU load and Event Loop lag with millisecond precision.
- **Dynamic Cost Scaling**: When system pressure is high (e.g., lag > 80ms), the cost of a request is automatically multiplied (up to 2.0x).
- **Capacity Boost**: During idle periods, request costs can be reduced (0.8x), rewarding users with higher throughput when resources are abundant.

### 3. Trusted Host Validation

To prevent SSRF and unauthorized configuration during the setup phase, the middleware enforces **Dynamic Host Trust**:

- **Local Trust**: Loops back (127.0.0.1) are always trusted for local admin access.
- **Origin Sync**: If `process.env.ORIGIN` is defined, only that specific host is allowed access to bootstrap routes.
- **Fail-Safe**: If no host is configured, only local access is permitted.

### 4. Integrated Performance Telemetry

The gatekeeper tracks system health metrics in real-time:

- **Init Duration**: Records how long the system takes to reach the `READY` state.
- **Security Violations**: Increments counters for untrusted host attempts or restricted route access.
- **Service Degradation**: Real-time heartbeat monitoring of DB, Cache, and Media adapters.
- **Adaptive Multiplier**: Records the current rate-limit multiplier based on hardware stress.

### 4. Incremental Service Refresh

The `reinitializeSystem` logic has been optimized to perform surgical updates rather than full destructive resets:

- **Config Hot-Reload**: Re-parses `private.ts` without dropping active database connections.
- **Dynamic Settings Sync**: Automatically re-synchronizes the in-memory settings cache with the database after configuration changes.
- **Zero-Downtime Transitions**: Minimizes the window where requests are blocked during system updates, ensuring high availability during administrative operations.

## 🔮 Future Improvements

1.  **Granular Sub-States**:
    - Instead of a global `INITIALIZING`, track `DB_CONNECTING`, `CACHE_WARMING`, `INDEX_BUILDING`.
    - Allow partial traffic (e.g., read-only) during specific sub-states.

2.  **Event-Driven Transitions**:
    - Move from polling/check-based transitions to an event bus architecture.
    - Example: `db:connected` event triggers transition to `CACHE_WARMING`.

3.  **Circuit Breakers**:
    - Automatically disable specific widgets or API endpoints if their dependent services fail, keeping the rest of the site `READY`.
