# Trash & Recovery API Reference

SveltyCMS implements a **Native Soft Delete** system. By default, content is moved to a "Trash" state rather than being permanently removed, allowing for accidental deletion recovery while maintaining database integrity.

---

## ⚡ Quick Start

| Feature         | HTTP Endpoint                          | Local SDK Equivalent                                             |
| :-------------- | :------------------------------------- | :--------------------------------------------------------------- |
| **Soft Delete** | `DELETE /api/collections/[id]/[entry]` | `locals.cms.delete(id, entry)`                                   |
| **Restore**     | `POST /api/trash/restore`              | `locals.cms.collections.update(id, entry, { isDeleted: false })` |
| **Hard Delete** | `DELETE ...?permanent=true`            | `locals.cms.delete(id, entry, { permanent: true })`              |

---

## 1. The Goal

Safely remove content from public view without data loss, or recover previously deleted items to their active state.

---

## 2. The Solution

### Soft Deleting Content

By default, all delete operations are "soft."

**Endpoint**: `DELETE /api/collections/posts/123`
**Mechanics**: Sets `isDeleted: true` and appends a `_DELETED_[timestamp]` suffix to unique fields (like slugs) to prevent index collisions.

### Restoring from Trash

**Endpoint**: `POST /api/trash/restore`
**Payload**:

```json
{
  "collectionId": "posts",
  "entryId": "123"
}
```

### Local SDK (Permanent Purge)

To bypass the trash and remove data forever (e.g., for GDPR compliance).

```typescript
await locals.cms.delete("posts", "123", { permanent: true });
```

---

## 3. The Mechanics

The Trash system uses a **Filter-at-Source** pattern to ensure deleted content never leaks into public queries.

```mermaid
graph TD
    A[Delete Request] --> B{Permanent?}
    B -- Yes --> C[Hard Delete from DB]
    B -- No --> D[Soft Delete]

    D --> E[isDeleted = true]
    E --> F[Mangle Unique Fields]
    F --> G[Invalidate Cache]

    subgraph "Query Layer"
    H[findMany / Search] --> I{Include Deleted?}
    I -- No (Default) --> J[Filter: {isDeleted: false}]
    I -- Yes --> K[Return All]
    end
```

### Unique Field Mangling

To prevent "Slug Collisions" (where a new post wants a slug used by a deleted one), SveltyCMS mangles unique fields:

- **Active**: `my-post`
- **Trashed**: `my-post_DELETED_1710793389`

> [!WARNING]
> **Restore Collisions**: If you attempt to restore an item but a new active item already exists with the same unique value (e.g., same slug), the restore will fail with `409 CONFLICT`. You must rename one of the items before restoring.
