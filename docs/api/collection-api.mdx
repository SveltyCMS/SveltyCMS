# Collection API Reference

The Collection API is the primary interface for **mutating** data within SveltyCMS. While reading data is prioritized via SvelteKit `load()` functions or the [Search API](./search-api.mdx), this API provides the critical endpoints for creating, updating, and managing the lifecycle of your content.

## 1. Architecture: The SSR-First Shift

In the 2026 SveltyCMS engine, **direct GET endpoints for listing collection entries have been removed** from the Collection API route.

### Why this change?

1.  **Eliminate Waterfall**: For the Admin Studio and standard frontend pages, data is fetched during SSR via `+page.server.ts`, preventing the "Double Fetch" pattern (Server -> Browser -> API -> Server).
2.  **Type Safety**: Native DB adapters provide full TypeScript intellisense that REST strings cannot.
3.  **Performance**: Sub-1ms hot-paths are achieved by querying the database directly within the SvelteKit request lifecycle.

### How to READ data now?

| Scenario                 | Recommended Solution                                                       | Latency   |
| :----------------------- | :------------------------------------------------------------------------- | :-------- |
| **SvelteKit App**        | Use [**Local SDK**](./local-vs-http-api.mdx) (`locals.cms.find`)           | **0–5ms** |
| **Headless (Next/Nuxt)** | Use the [**Search API**](./search-api.mdx) (filters & sorting)             | 50–150ms  |
| **Complex Queries**      | Use the [**GraphQL API**](./graphql-api.mdx) (query exactly what you need) | 50–200ms  |

> [!TIP]
> **Pro-Tip**: In SvelteKit `+page.server.ts` or `actions`, **always use `locals.cms`**. It bypasses the entire network stack, JSON serialization, and external auth overhead, making your app significantly faster and more secure.

## 2. Mutation Reference (Create, Update, Delete)

All mutations require an `Authorization: Bearer <token>` header and a valid `Content-Type: application/json`.

### Base Endpoint Pattern

`/api/collections/[collectionId]`

> [!NOTE]
> Use the **Internal ID** (e.g., `23d772dd3783492...`) for `[collectionId]`, which you can find in the Collection list or retrieve via the **Content Manager** (`contentManager.getCollectionById`).

### A. Create Entry (POST)

**Endpoint**: `POST /api/collections/[collectionId]`

**Payload**:

```json
{
  "title": "My New Post",
  "content": "Full article content...",
  "status": "published"
}
```

**Mechanics**:

- The system automatically assigns `_id`, `createdAt`, `updatedAt`, and `createdBy`.
- All data passes through the **Widget Validation System** before persistence.

### B. Update Entry (PATCH)

**Endpoint**: `PATCH /api/collections/[collectionId]/[entryId]`

**Payload**:

```json
{
  "status": "archived"
}
```

**Mechanics**:

- Performs a partial merge. Fields not included in the payload are preserved.
- Refreshes the `updatedAt` and `updatedBy` metadata.
- Automatically invalidates the page cache for this collection.

### C. Delete Entry (DELETE)

**Endpoint**: `DELETE /api/collections/[collectionId]/[entryId]`

**Query Params**:

- `permanent=true` (Optional): Forces a hard delete. Otherwise, entries are soft-deleted to the Trash.

## 3. Batch Operations

For managing multiple items (e.g., publishing 50 drafts at once), use the `/batch` endpoint to avoid multiple network roundtrips.

**Endpoint**: `POST /api/collections/[collectionId]/batch`

**Payload**:

```json
{
  "action": "status",
  "entryIds": ["id1", "id2", "id3"],
  "status": "published"
}
```

**Available Actions**:

- `status`: Update the publishing state.
- `delete`: Soft or hard delete the group.
- `clone`: Duplicate the targeted entries.

## 4. Security & Multi-Tenancy

### Automatic Tenant Scoping

If `MULTI_TENANT` is enabled in your `private.ts` config, the API **automatically** scopes every request:

- **Write**: New entries are stamped with the `tenantId` from the authenticated user.
- **Access**: Attempting to PATCH or DELETE an entry belonging to a different tenant returns a `404 NOT_FOUND` for security obfuscation.

### Permission Gates

Endpoints check permissions sequentially:

1.  **Auth Check**: Is the request signed with a valid JWT?
2.  **Role Check**: Does the user's role permit `write` or `delete` on this specific collection?
3.  **Widget Check**: Are there field-level restrictions (e.g., "Read-Only" fields) defined in the widget schema?

## 📡 Response Shape

All Collection API responses include performance telemetry.

```json
{
  "success": true,
  "data": { ... },
  "performance": {
    "duration": 4.5,
    "dbTime": 2.1
  }
}
```

**Error Codes**:

- `COLLECTION_NOT_FOUND` (404)
- `VALIDATION_FAILED` (400) — Check `details` for field-specific errors.
- `UNAUTHORIZED` (401) — Token missing or expired.
