# Collection API Reference

The Collection API is the primary interface for **mutating** and **reading** data within SveltyCMS. While the system prioritizes server-side performance using [**Local SDK**](./local-vs-http-api.mdx), this API provides the critical dispatcher for external consumers (including headless frontends and third-party integrations).

---

## 📡 Modern Dispatcher Architecture

As of version 2026.4, the Collection API uses a **Modular Dispatcher** structure. This provides:

- **Consistent Response Shapes**: Unified success/error JSON.
- **304 Support**: Built-in ETag-based caching for high-performance GET requests.
- **Improved Security**: Enhanced CSRF and Multi-tenant verification.

---

## 🏗️ Querying & Mutations

All requests require an `Authorization: Bearer <token>` header and set `Content-Type: application/json`.

### 1. Read Operations (GET)

While we recommend [**Local SDK**](./local-vs-http-api.mdx) for SSR, these endpoints are fully available for external apps.

#### A. List Collections

**Endpoint**: `GET /api/collections`  
**Description**: Returns a list of all authorized schemas/collections.  
**Query Params**:

- `includeFields=true`: Include full schema definitions.
- `includeStats=true`: Include item counts (performance-heavy logic).

#### B. Search & Query entries

**Endpoint**: `GET /api/collections/search`  
**Query Params**:

- `q`: Search string in indexed fields.
- `collections`: Comma-separated list of IDs to filter by.
- `limit`/`page`: Standard pagination.
- `filter`: JSON-stringified Mongo-style filter.

---

## 🛡️ Response Shape

All SveltyCMS API responses follow a unified structure to simplify frontend integration.

### Success Response (200 OK)

Standard responses wrap results in a `data` property.

```json
{
  "success": true,
  "data": { "title": "My Post", "id": "..." }
}
```

### Not Modified (304)

The API automatically sends `etag` headers. If the client sends an `If-None-Match` header that matches the current data hash, the server returns a **304** with NO body, saving significant bandwidth.

### Error Response (4xx/5xx)

Errors follow a consistent structure via the Unified Error Handler.

```json
{
  "success": false,
  "message": "Authentication required",
  "error": {
    "code": "UNAUTHORIZED",
    "status": 401
  }
}
```

## 🛠️ Mutation Reference

### A. Create Entry (POST)

**Endpoint**: `POST /api/collections/[collectionId]`  
**Payload**: JSON object matching the collection schema.  
**Permission**: `collections:create`

### B. Update Entry (PATCH)

**Endpoint**: `PATCH /api/collections/[collectionId]/[entryId]`  
**Mechanics**: Performs a partial merge. Only include the fields you wish to change.

### C. Delete Entry (DELETE)

**Endpoint**: `DELETE /api/collections/[collectionId]/[entryId]?permanent=true`

---

## 📦 Batch Operations

For large-scale changes, use the `/batch` endpoint to avoid multiple roundtrips.

**Endpoint**: `POST /api/collections/[collectionId]/batch`  
**Payload**:

```json
{
  "action": "status",
  "entryIds": ["id1", "id2"],
  "status": "published"
}
```
