# 🚀 SveltyCMS Performance Benchmarks

> [!IMPORTANT]
> **EU Compliance**: Competitive performance comparisons in this document are based on publicly available documentation as of June 2026. All SveltyCMS metrics are self-measured via reproducible benchmark suites (`bun test tests/benchmarks/`). Competitor figures are derived from publicly documented framework overhead, not side-by-side benchmarks. See [Competitive Comparison](../competitive-comparison.mdx) for full methodology.
>
> **Recorded** benchmark runs update their report section immediately. Normal `bun test` runs print to console only unless `BENCHMARK_RECORD=1` is set.
>
> **Intelligence Features**: Cross-test correlation, adaptive budgets, forecasting, fix suggestions, differential execution, sub-component timing, memory trend tracking.

## 🔬 How a Recorded Benchmark Pushes Data

````bash
BENCHMARK_RECORD=1 bun test tests/benchmarks/auth-performance.test.ts
```text

> [!NOTE]
> Without `BENCHMARK_RECORD=1`, benchmarks are console-only — no files are modified.
> The full matrix runner (`scripts/benchmark-matrix/index.ts`) always records.

### Record Modes

| Mode      | Trigger                             | SQLite |        MDX        | Executive Summary | Console trend comparison |
| --------- | ----------------------------------- | :----: | :---------------: | :---------------: | :----------------------: |
| `none`    | Normal `bun test`                   |   ❌   |        ❌         |        ❌         |       ✅ read-only       |
| `history` | `BENCHMARK_HISTORY_ONLY=1`          |   ✅   |        ❌         |        ❌         |            ✅            |

### Data Flow

```
flowchart TD
    A["bun test auth-performance.test.ts"] --> B["setupBenchmarkServer"]
    B --> C["runBenchmark x N scenarios"]
    C --> D["exportResult per scenario"]
    D --> E["reportBenchmark(result, opts)"]
    E --> F1["Update section label with trend"]
    E --> F2["Append root cause insight"]
    E --> F3["Update Executive Summary"]

    C --> G["printTruthTable"]
    G --> H["pushTableToMdx (bridge)"]
    H --> I["Tag discovery -> find correct section"]
    I --> J["Write truth table between tags"]

    G --> K["printSummaryTable"]
    K --> L["appendSummaryToMdx (bridge)"]

    J --> M["benchmark_sqlite.mdx"]
    L --> M
    F1 --> M
    F2 --> M
    F3 --> M
```text

### Step-by-step

| Step | Function                        | What happens                                                                                                                                                                                                                                                                                              |
| ---- | ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | `setupBenchmarkServer()`        | Starts a SvelteKit server on a random port with `PRESET=demo`. Setup wizard seeds admin/roles/settings + 11 preset collections via `seedPresetCollections()`. Then `setup-benchmarks.ts` seeds entry data via real HTTP endpoints. Uses shared `benchmark_shared` database, skips seeding if data exists. |
| 2    | `runBenchmark()`                | Runs the test scenario N times, collects timing data                                                                                                                                                                                                                                                      |
| 3    | `exportResult()`                | Saves JSON to `results/<db>/`, appends to `history.jsonl` (debug log), triggers trend analysis                                                                                                                                                                                                            |
| 4    | `reportBenchmark(result, opts)` | Persists to `history.sqlite` (canonical), runs trend analysis, updates MDX section                                                                                                                                                                                                                        |
| 5    | `printTruthTable()`             | Builds ASCII table -> calls `pushTableToMdx()` bridge                                                                                                                                                                                                                                                     |
| 6    | `pushTableToMdx()`              | Compatibility bridge — discovers the correct `<!-- TAG_TABLE_START -->` section, writes truth table                                                                                                                                                                                                       |
| 7    | `printSummaryTable()`           | Builds summary table -> calls `appendSummaryToMdx()` bridge                                                                                                                                                                                                                                               |

### Guardrails

| Guard                      | Behavior                                                                                                    |
| -------------------------- | ----------------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| **`BENCHMARK_RECORD=1`**   | Gates all MDX writes. Without it, benchmarks are console-only. Matrix always records.                       |
| **Single-test confidence** | Root cause is marked "suspected" — cross-test correlation checks `history.sqlite` for related test patterns |
| **Partial-run watermark**  | `📋 Partial update: auth-performance / SQLITE / 2026-05-30` in Executive Summary                            |
| **Stable IDs**             | Each run keyed as `testId/db/redis-on                                                                       | off/phase/metric` for durable trend history |
| **Per-DB budgets**         | SQLite/PG/MariaDB/MongoDB each have different cold/warm/CRUD/cache/concurrency budgets                      |

## 🧠 Smart Features

| Feature                       | Description                                                                                                              |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **Record Mode Gates**         | `BENCHMARK_RECORD=1` enables MDX writes; normal `bun test` is console-only. Matrix always records.                       |
| **Instant MDX Updates**       | Each recorded `bun test` writes truth table + trend + insight directly to its report section                             |
| **Rolling Trend Baseline**    | Compares current run against median of recent runs from `history.sqlite` (canonical)                                     |
| **Severity Thresholds**       | ⚪ <5% stable, 🟡 5-10% watch, 🟠 10-20% warning, 🔴 >20% regression                                                     |
| **Multi-Metric Tracking**     | Label shows `avg`, `p95`, and `rps` deltas: `🔴 avg +35% p95 +22% (12 runs)`                                             |
| **Root Cause Classification** | 7 categories: normal variance, adapter bottleneck, cold start, GC pause, improvement, throughput drop, severe regression |
| **Confidence Levels**         | Single-test = "suspected", cross-correlation match = "confirmed", low samples = "watch"                                  |
| **Code Path Recommendations** | Each regression links to specific source files to investigate                                                            |
| **Executive Summary Alerts**  | Regression alerts appear at the top of the report with severity ranking                                                  |
| **Partial-Run Watermark**     | "Partial update" marker when not all tests ran                                                                           |
| **Tag Discovery**             | Matches test filename to MDX section via prefix/substring/bidirectional matching                                         |
| **Atomic MDX Writes**         | Temp file + rename prevents half-written reports                                                                         |
| **Idempotent SQLite**         | `INSERT OR IGNORE` prevents duplicate rows on retry                                                                      |
| **Per-DB Budgets**            | SQLite/PG/MariaDB/MongoDB each have different cold/warm/CRUD/cache/concurrency budgets                                   |
| **Shared Database**           | All benchmarks reuse `benchmark_shared.sqlite` — first run seeds, subsequent runs skip seeding (smart detect)            |
| **Persistent Data**           | Database file persists between runs — no cleanup on stop. Write benchmarks recreate their own collections as needed      |
| **58 Enhanced Benchmarks**     | 12 dimensions: Baseline, Adapter, Internals, Logic, API, Scale, Resilience, Security, Governance, Soak, Streaming, Failover |

## 📝 Creating a New Benchmark

Every benchmark test MUST follow this structure to integrate with the reporting pipeline:

### Required structure

```
/**
 * @file tests/benchmarks/my-benchmark.test.ts
 * @description One-line description of what this benchmark measures.
 */
import {
  test,
  runBenchmark,
  exportResult,
  printTruthTable,
  printSummaryTable,
  getDbType,
} from "./modules/benchmark-utils";
import "../unit/bun-preload.ts";

test("My Benchmark Name", async () => {
  const server = await setupBenchmarkServer();
  const baseUrl = server.baseUrl;

  const result = await runBenchmark({
    name: "Scenario Name",
    iterations: 100,
    concurrency: 4,
    onIteration: async (i) => {
      await fetch(`${baseUrl}/api/...`);
    },
  });

  exportResult(result);

  printTruthTable({
    title: "SVELTYCMS — MY BENCHMARK AUDIT",
    shortLabel: "MyBench",
    results: [{ ...result, name: "Scenario Name" }],
  });

  printSummaryTable([
    { key: "Latency", val: result.avgMs, unit: "ms" },
    { key: "Throughput", val: result.rps, unit: "req/s" },
  ]);
}, 120000);
```text

### Required checklist

- [ ] `@file` and `@description` header comments
- [ ] Import from `./modules/benchmark-utils`
- [ ] Call `setupBenchmarkServer()` (if server needed) or standalone setup
- [ ] Call `runBenchmark()` with `name`, `iterations`, `concurrency`, `onIteration`
- [ ] Call `exportResult(result)` for EACH scenario (triggers trend + history)
- [ ] Call `printTruthTable()` with `title`, `shortLabel`, `results`
- [ ] Call `printSummaryTable()` with array of `{key, val, unit}`
- [ ] Wrap in `test("name", async () => {...}, timeout)`
- [ ] Register in `scripts/benchmark-matrix/benchmark-scripts.ts` with `path`, `label`, `shortLabel`, `section`

### Tag matching rules

The `pushTableToMdx` bridge discovers the correct MDX section by:

1. Getting the test filename from the call stack (e.g., `my-benchmark.test.ts` -> base `my_benchmark`)
2. Scanning all `<!-- TAG_TABLE_START -->` tags in the MDX
3. **Preferring tags after `<!-- BENCHMARK_END -->`** (educational sections over matrix duplicates)
4. Matching via: prefix match, substring match, or bidirectional match
5. Falling back to `shortLabel` from `printTruthTable`

## 🔬 How to Run

> [!TIP]
> Benchmarks share a single `benchmark_shared.sqlite` database.
> First run seeds all data. Subsequent runs detect existing data and skip re-seeding — **much faster**.
> To force a fresh seed, delete `config/database/benchmark_shared.sqlite`.

**Console only (safe for normal dev):**

```
bun test tests/benchmarks/auth-performance.test.ts
```text

**All benchmarks (sequential, with recording):**

```
bun run scripts/run-all-benchmarks.ts
```text

> [!NOTE]
> The runner runs each test sequentially and reports pass/fail.

**Single test (record to report):**

```
BENCHMARK_RECORD=1 bun test tests/benchmarks/auth-performance.test.ts
```text

**History only (no MDX mutation, useful in CI):**

```
BENCHMARK_HISTORY_ONLY=1 bun test tests/benchmarks/auth-performance.test.ts
```text

**Full SQLite matrix:**

```
bun run scripts/benchmark-matrix/index.ts --sql
```text

**Differential (only tests affected by recent changes):**

```
bun run scripts/benchmark-matrix/index.ts --sql --differential
```text
````
