# Testing Strategy: Three-Layer Architecture

SveltyCMS uses a **three-layer testing pyramid** where each layer is optimized for what it does best. No layer duplicates another — pure algorithms get unit tests, API contracts get integration tests, and user journeys get E2E tests.

## Layer Boundaries

| Layer         | Scope       | Tooling                    | Tests What                                                              | Doesn't Test                    |
| ------------- | ----------- | -------------------------- | ----------------------------------------------------------------------- | ------------------------------- |
| **White-Box** | Unit        | Vitest (jsdom)             | Pure algorithms, schemas, data transforms, state machines               | I/O, DB, filesystem, network    |
| **Black-Box** | Integration | `run-integration-tests.ts` | Real server + real DB — API contracts, middleware, RBAC, DB agnosticism | UI, browser rendering           |
| **E2E**       | Browser     | Playwright                 | User journeys, accessibility, keyboard nav, visual flows                | Internal implementation details |

---

## 1. White-Box Testing (Unit)

**"Test pure logic knowing how it works."**

Files: `tests/unit/**/*.test.ts` | Runner: `bun run test:unit` | Tests: 915

These tests have full access to internal code structure. They're fast (milliseconds), parallelizable, and catch logic regressions immediately.

**What belongs here:**

- Pure functions (date formatting, crypto, validation schemas)
- Data transforms (tree adapters, logic builders, pluralization)
- Svelte 5 rune stores (state transitions, derived values)
- Widget validation schemas (Valibot, defaults, edge cases)
- Security utilities (Argon2id, CSPRNG tokens, SHA-256 checksums)

**What does NOT belong here:**

- Database adapter calls (mock the adapter, test at integration layer)
- Filesystem operations (real filesystem = integration test)
- Cloud SDK wrappers (S3, Cloudinary — integration/E2E only)
- Form submission flows (use E2E Playwright)

---

## 2. Black-Box Testing (Integration)

**"Test the API as a consumer sees it."**

Files: `tests/unit/api/*.test.ts` | Runner: `bun run test:integration` | Tests: ~450

The integration runner (`scripts/run-integration-tests.ts`) orchestrates the full lifecycle:

1. Generates `config/private.test.ts` with isolated DB credentials
2. Builds the production server (`bun run build`)
3. Starts preview server on a free port
4. Seeds the database via `/api/testing`
5. Runs tests against real HTTP endpoints
6. Tears down the server and cleans up

**What belongs here:**

- API endpoint contracts (REST, GraphQL)
- Authentication flows (login, session, 2FA, token rotation)
- RBAC enforcement (admin/editor/viewer permission boundaries)
- Database CRUD across all 4 adapters (SQLite, MongoDB, PostgreSQL, MariaDB)
- Middleware chain verification (firewall, rate-limit, auth, CSP headers)
- Media upload/download with SSRF protection

**Key hardening:**

- **Fail-Closed by default**: Unmapped API routes return 403
- **No backdoors**: Tests authenticate via real login, not header bypasses
- **Database agnostic**: Same suite runs against all 4 databases
- **Isolation**: Each test run uses its own temp database, never touches production

---

## 3. E2E Testing (Browser)

**"Test the full user journey in a real browser."**

Files: `tests/e2e/*.spec.ts` | Runner: `npx playwright test` | Scenarios: ~40

Playwright drives Chromium, Firefox, and WebKit against the running server. These tests verify that UI, accessibility, and visual flows work end-to-end.

**What belongs here:**

- Setup wizard flow (all 4 databases)
- Login → Dashboard → Content creation → Publish
- RBAC enforcement in the UI (admin sees X, editor doesn't)
- Media library interactions (upload, search, share, version history)
- Keyboard navigation and focus traps
- Accessibility audits (Axe-Core, RTL mirroring)
- Collection builder drag-and-drop

**What does NOT belong here:**

- API contract validation (use integration tests — faster, more reliable)
- Database adapter verification (use integration tests — covers all 4 DBs)
- Pure algorithm edge cases (use unit tests — milliseconds vs seconds)

---

## 4. Benchmark Audit

**"Prove performance under stress."**

Runner: `bun run scripts/benchmark-matrix/index.ts` | Metrics: 50+ across 4 databases

Not a functional test layer — this validates SRE-grade resilience:

- Cold start time (< 1s target)
- REST p95 latency (< 0.6ms SQLite, < 0.7ms others)
- Chaos resilience (500ms DB brownouts)
- Migration throughput (> 6,600 entries/s)
- Index pressure (100,000+ entries with complex filters)

## Layer Selection Guide

| If you're testing...                      | Use this layer                                    |
| ----------------------------------------- | ------------------------------------------------- |
| A validation schema (Valibot)             | Unit — `safeParse()` with edge cases              |
| A date formatting function                | Unit — fast, deterministic                        |
| An API endpoint returning JSON            | Integration — real HTTP, real DB                  |
| Whether an editor can access admin routes | Integration — RBAC through real middleware        |
| The setup wizard flow                     | E2E — Playwright fills real forms                 |
| Keyboard tab order through a modal        | E2E — Playwright `page.keyboard.press("Tab")`     |
| File upload with SSRF protection          | Integration — real server, real file I/O          |
| S3 upload actually works                  | Integration — real SDK call                       |
| A CSS class changes on hover              | Not worth testing — visual regression if critical |
| A store's derived value computation       | Unit — Svelte 5 runes, in-memory                  |

## Continuous Integration

```bash
# Local pre-commit (fast path — related tests only)
bun run scripts/quality-gate.ts

# Full CI pipeline (all 4 databases)
# - Unit: Vitest (915 tests, ~15s)
# - Integration: SQLite → MongoDB → PostgreSQL → MariaDB
# - E2E: Playwright (Chromium, Firefox, WebKit)
# - Benchmarks: On main branch only
```
