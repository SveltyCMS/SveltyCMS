# Testing Strategy: The Hybrid Pyramid

SveltyCMS employs a pragmatic, modern testing strategy that leverages the strengths of both **Black-Box** and **White-Box** testing methods. This hybrid approach ensures we deliver a secure, database-agnostic CMS without sacrificing developer velocity or code quality.

## The Dual Strategy

We use a layered approach where each method is applied to the domain where it shines most:

| Method        | Scope                 | Tooling                               | Focus                                                                                  |
| :------------ | :-------------------- | :------------------------------------ | :------------------------------------------------------------------------------------- |
| **White-Box** | **Unit Tests**        | **Vitest**                            | Logic correctness, edge cases, internal consistency, **Target: 100% Branch Coverage**. |
| **Black-Box** | **Integration & E2E** | **Vitest / Playwright**               | Real-world behavior, security posture, API contracts, Fail-Closed RBAC, DB agnosticism.|

### 1. White-Box Testing (Unit)

**"Test the code knowing how it works."**

We use **Vitest** for white-box unit testing. These tests presume full knowledge of the internal code structure, classes, and logic paths. Following our recent optimization, we have consolidated our unit suite into **~670 highly performant tests** in `tests/unit/`.

**What we test here:**

- **Pure Functions & Utilities**: Formatters, validators, crypto helpers.
- **Services**: Business logic that doesn't depend on external I/O (or uses mocks).
- **Svelte Stores**: State management logic and reactivity via Runes.
- **Widgets**: Component logic and validation schemas.

**Why White-Box?**

- **Speed**: Vitest unit tests run in milliseconds (total suite < 10s).
- **Thoroughness**: We can hit every `if/else` branch and edge case that might be hard to trigger via an API call.
- **Stability**: Refactoring internal implementation details might break these tests, which is good—it ensures the _unit_ still behaves correctly in isolation.

### 2. Black-Box Testing (Integration & E2E)

**"Test the system as a consumer sees it."**

We use **Black-Box API testing** for all integration and end-to-end scenarios. These tests treat the entire running server as an opaque box.

**What we test here:**

- **API Endpoints**: REST and GraphQL request/response cycles via the **Fail-Closed Dispatcher**.
- **Authentication & Authorization**: Real login flows, token exchanges, and mandatory RBAC enforcement.
- **Database Operations**: CRUD actions across different database adapters (MongoDB, SQLite, PostgreSQL).
- **Middleware Chains**: Ensuring hooks (firewall, rate-limit, auth) execute in the correct order.

**Why Black-Box?**

- **Security**: Tests must pass through real security guards. There are no "test backdoors" or mocked permission checks.
- **Fail-Closed Enforcement**: All API requests are verified against `ENDPOINT_PERMISSIONS`. Unmapped routes are blocked by default.
- **Database Agnosticism**: The test suite doesn't care if the backend is SQLite or PostgreSQL; it only cares about the JSON response. This allows us to run the exact same suite against all supported databases.
- **Production Parity**: Tests run against a production-like build (`vite preview`), catching issues related to headers, serialization, and SSR that unit tests miss.
- **Resilience**: Internal refactors (e.g., changing from Mongoose to Drizzle) generally don't break these tests as long as the API contract remains the same.

## Beyond Functional: The Road to 100% Confidence

To truly aim for "100%" system reliability, we go beyond simple functional tests:

### 3. Static Analysis & Type Safety (Continuous)

- **TypeScript**: Strict mode enabled (`strict: true`) for 100% type safety.
- **oxlint / oxfmt**: Enforce consistent code style and best practices with ultra-high performance (<100ms for the entire project).
- **Svelte Check**: Validates Svelte templates and reactivity graph (Runes).

### 4. Advanced Verification (Future Scope)

To reach the "100%" ambition, we are planning:

- **Mutation Testing (Stryker)**: "Testing the tests" by introducing random bugs to ensure tests catch them.
- **Fuzz Testing**: Throwing random/garbage data at API endpoints to find crashes/leaks.
- **Visual Regression**: Pixel-perfect UI verification.
- **Dependency Auditing**: Automated `npm audit` checks in CI.

## Continuous Integration (CI) Architecture

SveltyCMS uses a performance-optimized CI pipeline designed for maximum reliability and fast feedback.

### 1. Parallel Quality Gate

All static analysis and unit checks run in parallel:

- **Lint & Type Check**: oxlint/oxfmt linting + `svelte-check` validation.
- **Unit Tests & Coverage**: Unified Vitest suite in `tests/unit/` with Codecov reporting.
- **Build Verification**: Production SSR and Client build.

### 2. Consolidated Database Matrix

Integration and E2E tests are grouped by database engine (MongoDB, MariaDB, PostgreSQL, SQLite).

- **One Leg, Dual Suite**: Each matrix leg starts its database service once and runs both Integration and E2E tests sequentially.
- **Cross-Browser Coverage**: E2E tests validate against Chromium, Firefox, and Webkit.

## Summary

By combining **Vitest-powered White-Box unit tests** for speed and logic depth with **Black-Box API testing** for Fail-Closed security and integration reliability, SveltyCMS achieves a robust quality assurance posture. our goal is **100% System Confidence**.
