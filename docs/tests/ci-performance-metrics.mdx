# Bundle Size Analysis & Performance Metrics (April 2026)

We actively monitor our application bundle size and API latency to prevent performance regressions. Our target is **sub-10ms database persistence** and **sub-1s cold starts**.

## 📊 Enterprise Performance Matrix

The **Enterprise Benchmark Matrix (Audit 2.0)** is our primary tool for cross-database performance validation. As of April 19, 2026, the orchestrator has been deeply stabilized to support massive concurrent loads across all 4 primary database engines (**SQLite, MongoDB, PostgreSQL, and MariaDB**) with 100% passing metrics.

It measures:

- **REST API Latency**: E2E collection throughput using the unified Dispatcher.
- **GraphQL Resolvers**: Efficiency, AST depth limits, and N+1 resolution.
- **Relational Performance**: JOINs, population, and deeply nested relations (Depth 2-3).
- **Middleware Overhead**: Microsecond-latency added by Security, Auth, and RBAC layers.
- **Realtime Latency**: WebSocket connection and broadcast timing.
- **Memory Stability**: RSS and Heap growth detection across sustained 180s loads.

### 🚀 Running the Orchestrator

The orchestrator code is modularized inside `scripts/benchmark-matrix/` for ultimate maintainability, but is easily executed via the main barrel file:

```bash
# Run the full matrix across all 4 databases and 8 variants (including Redis)
bun run scripts/benchmark-matrix/index.ts

# Run only a specific database
bun run scripts/benchmark-matrix/index.ts --db=postgresql,mariadb

# Run a specific benchmark section
bun run scripts/benchmark-matrix/index.ts --db=sqlite --section=api
```

> [!NOTE]
> Running the full matrix requires active local instances (or Docker containers) of MongoDB (`:27017`), PostgreSQL (`:5432`), and MariaDB (`:3306`). The orchestrator will automatically provision the required isolated `_global` and `_setup` databases for the test.

### Retention & Trend Analysis (SQLite)

The matrix maintains a specialized retention policy to balance historical context with actionable trends:

- **SQLite Backend**: All results are persisted in `history.sqlite` for professional time-series analysis.
- **Trend Window**: Mermaid charts and trend indicators (🟢/🔴) analyze the **last 5 runs**.
- **Historical Depth**: Legacy `history.json` is maintained for PR diffs, persisting up to **20 runs** per database variant.

## 🔗 Relational Performance Testing

With the 2026 update, we've introduced specialized relational benchmarks to ensure that complex data structures remain performant:

- **GQL Nested**: Tests Depth 2-3 relationship resolution.
- **REST Relational Search**: Validates filtering by related fields via the aggregation engine.

**File**: `tests/benchmarks/relational-performance.test.ts`

## Running Analysis Locally

To check the bundle size and basic performance locally:

```bash
bun run build:stats
```

This generates detailed reports including:

- Size of each chunk (raw, gzipped, brotli).
- Comparison with the previous build size.
- Warnings reported if predefined performance budgets are exceeded.
- An exported `bundle-report.json` designed for automated CI consumption.

## 🧩 Widget Performance Auditing (Core & Custom)

As part of the **Enterprise Matrix**, SveltyCMS performs a specialized overhead pass for widgets. This ensures that custom widgets do not introduce "Silent Regressions" into the core API.

### The Audit Process:

1. **Definition Pillar Profiling**: Measures the latency added by a widget's `modifyRequest` and `transform` logic in the server-side pipeline.
2. **Aggregation Stress Test**: Verifies that custom `listSchemas` and search aggregations remain efficient as database sizes grow.
3. **Core Baseline**: All custom widgets are measured against the **Input (Core)** baseline (~0.01ms overhead).

> [!WARNING]
> Custom widgets exceeding a **2ms per-field overhead** are flagged in the matrix as "Performance Hazards" and should be optimized using the **Vectorized Processing** patterns.
