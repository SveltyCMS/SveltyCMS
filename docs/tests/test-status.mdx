## 1. Black-Box Integration Tests (`scripts/run-integration-tests.ts`)

| Metric                      | Value                                                   |
| --------------------------- | ------------------------------------------------------- |

**Note**: Integration test pass rate varies by database adapter. SQLite passes the majority; MariaDB/Mongo/Postgres expose auth/permission differences tracked as separate bugs. The test config safety guard (`scripts/check-test-db-safety.ts`) prevents the most common failure mode: running tests against a non-isolated (production) database.

**Coverage**: 48 spec files across 6 domains

| Domain              | Files | What's Tested                                                                                    |

| Security            | 4     | Defense-in-depth, negative scenarios, rate-limiting, XSS/SQLi                                    |
| Database (existing) | 7     | Contract tests, adapter-specific (SQLite/Mongo/Maria/Postgres), resilience                       |
| **Database (new)**  | **9** | **Parity matrix, error contract, cache contract, transactions, bulk ops, advanced CRUD, health** |
| Setup & SDK         | 2     | Setup wizard, LocalCMS SDK                                                                       |

**Hardening applied**:

- Security rate-limiter reset between tests → zero cross-test leakage
- `IDLE` state accepted in health checks → eliminates 10s settle delay
- Widget auto-registration → fixes 404 for valid widgets in fresh DB
- All caches + state stores cleared per-test → true isolation
- **Test DB safety guard** (`scripts/check-test-db-safety.ts`) runs in pre-commit and pre-push hooks → blocks unsafe `config/private.test.ts` before tests start
- **Shared classifier** (`src/utils/test-db-safety.ts`) used by server, integration runner, and git hooks → single source of truth for isolated DB name validation
- **Secret synchronization** — integration runner writes `TEST_API_SECRET` to `tests/e2e/.auth/test-secret.txt` before server start → eliminates 401 mismatch
- **`SKIP_COLLECTION_COMPILE`** flag prevents @src alias resolution failures during integration test builds on Windows

These measures ensure zero cross-test contamination and deterministic test results.

**Hardening — Benchmark & Integration Infrastructure**:

`tests/unit/benchmarks/benchmark-harness.test.ts` — 11 harness self-tests:

| Test file                 | Tests | Covers                                                                           |
| ------------------------- | ----- | -------------------------------------------------------------------------------- |
| benchmark-harness.test.ts | 11    | `assertSuccess` (7 cases), warmup isolation regression (2), error visibility (2) |

> **Why**: A MongoDB E11000 duplicate-key bug survived 3,600 silently-failed benchmark iterations because: (a) warmup errors were swallowed by an empty `catch {}`, (b) benchmark callbacks never checked `result.success`, and (c) no unit tests existed for the benchmark harness itself. These 11 tests validate that the framework correctly detects and reports failures at every layer.

`tests/integration/databases/adapter-parity.test.ts` — 15 cross-adapter CRUD parity tests:

| Test file              | Tests | Covers                                                                                      |
| ---------------------- | ----- | ------------------------------------------------------------------------------------------- |
| adapter-parity.test.ts | 15    | Insert, findOne, findMany, update, delete, count, upsert — identical assertions per adapter |

> **Why**: Each adapter had its own integration test file testing _different_ things. No unified suite ran the identical operation against all 4 adapters and asserted identical results. This test catches adapters that silently upsert where others throw, return different data shapes, or violate the `DatabaseResult` contract.

`tests/integration/databases/error-contract.test.ts` — 11 error handling contract tests:

| Test file              | Tests | Covers                                                                               |
| ---------------------- | ----- | ------------------------------------------------------------------------------------ |
| error-contract.test.ts | 11    | Duplicate keys, missing records, invalid inputs, error shape consistency per adapter |

> **Why**: SQL adapters silently wrap errors via `BaseAdapter.wrap()` (returning `{ success: false }`), while MongoDB catches `E11000` in `crud-methods.ts`. Both return `{ success: false }` but with different shapes and logging behavior. This test enforces consistent error handling across all 4 adapters.
