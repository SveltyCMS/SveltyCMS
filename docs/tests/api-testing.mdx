# API Testing & Coverage Report

SveltyCMS employs a rigorous **Black-Box Testing** strategy for its API layer, ensuring that all endpoints are verified for security, performance, and database agnosticism. All requests are managed by a central **Fail-Closed Dispatcher**.

> [!NOTE]
> This document consolidates our coverage reporting and historical testing summaries into a single source of truth.

## 🛡️ Test Isolation & Safety

SveltyCMS is designed for maximum safety. Running tests will **never** influence or overwrite your live production data. This is achieved through three layers of isolation:

1.  **Filename Isolation**: Integration tests use `sveltycms_test.db` (configured in `private.test.ts`), while production uses `test.db` (or your configured `DB_NAME`).
2.  **Environment Isolation**: Tests require `TEST_MODE=true`. This flag switches the configuration from `private.ts` to `private.test.ts` and changes the database path to `./config/database`.
3.  **API Gatekeeping**: The God-Mode `/api/testing` endpoint is hard-coded to be **inactive** unless the server is explicitly started with the `TEST_MODE` flag. In a production build, these endpoints are automatically disabled.
4.  **Network Lockdown**: Tests are restricted to `127.0.0.1` to ensure consistent behavior across different Operating Systems.

## 🏗️ Testing Architecture

Our API tests operate at the **Integration** level. Unlike unit tests, these interact with a fully running server instance, verifying the entire middleware chain (Firewall → Auth → Logic → DB).

### Key Components

1.  **Fail-Closed Gatekeeper (`+server.ts`)**: Central dispatcher that enforces mandatory authorization for EVERY endpoint. Unmapped routes are denied by default (403 Forbidden).
2.  **God-Mode API (`/api/testing`)**: Allows tests to programmatically reset the database and seed specific fixtures without direct DB access.
3.  **Mock Isolation**: Integration tests use real databases (isolated per run) while Unit tests use `tests/unit/setup.ts` for deep internal mocking.
4.  **Authentication Helpers**: Programmatic login and session management via `tests/unit/api/rbac.test.ts`.

## 🏛️ Historical Context & Evolution

Initially, SveltyCMS faced issues with error status codes (returning 500 instead of 401) and complex OAuth mocks. Through our **Fail-Closed Gatekeeper** and specialized integration runner, we have achieved:

- **Fail-Closed Authorization**: All routes are now protected by a central gatekeeper, returning 401/403 properly.
- **Isolated State**: The introduction of the `/api/testing` endpoint eliminated test interference.
- **Infrastructure Stability**: Resolved mocking conflicts in core services via Dependency Injection, achieving a 100% pass rate.
- **Ghost Caching Velocity**: Reduced SDK Read latency by **98.8%** (~0.612ms -> **0.007ms**), ensuring internal lookups are near-instant.
- **Distributed Consistency**: Integrated **Redis Pub/Sub** invalidation bridge for real-time cache parity across global nodes.
- **Performance**: Consolidated redundant unit tests and built new suites (727 tests) while maintaining 100% logic coverage, reducing suite latency.
