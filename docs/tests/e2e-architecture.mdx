# Black-box E2E – How the layers connect

SveltyCMS uses a **Parallel Worker Isolation** strategy to achieve high-concurrency E2E testing without database locks.

## ⚡ Database-per-Worker Isolation

To maximize performance in April 2026, we standardized on header-based database routing:

1.  **Header Propagation**: Playwright includes an `x-test-worker-index` header in every request.
2.  **Dynamic Connection Map**: The SQLite adapter parses this index and maintains a map of dedicated worker-specific files (e.g., `cms_worker_1.db`, `cms_worker_2.db`).
3.  **Zero Contention**: Each worker operates on its own physical file, allowing us to run **4+ parallel workers** with 100% isolation.

````mermaid
sequenceDiagram
    participant CI
    participant PW as Playwright (Worker 1)
    participant DB as SQLite (worker_1.db)
    participant App as Preview Server

    CI->>PW: npx playwright test --workers=4
    Note over PW,App: Worker 1 sends x-test-worker-index: 1

    PW->>App: GET /api/system/health
    App->>DB: Check connection for worker_1
...

    alt Wizard project (runs once per shard)
        PW->>App: Visit / → detect wizard UI
        App-->>PW: Render setup steps
        PW->>App: Fill DB choice + admin credentials
        PW->>App: Submit → system marked installed
    end

    PW->>PW: Run auth.setup.ts (dependency)
    Note over PW: /api/testing/reset (safety)
    PW->>App: Seed minimal admin if needed
    PW->>App: Login admin → save admin.json

    PW->>App: POST /api/invite → editor token
    PW->>App: POST /api/invite → author token

    alt Option A – fast (API assisted)
        PW->>App: POST /api/testing/signup-with-token editor
        PW->>App: POST /api/testing/signup-with-token author
    else Option B – realistic black-box
        PW->>App: Visit /signup?token=editor → fill form → submit
        PW->>App: Visit /signup?token=author → fill form → submit
    end

    PW->>PW: Save editor.json + author.json

    loop For each spec file
        PW->>PW: Use storageState: admin.json / editor.json / ...
        PW->>App: Test dashboard, content creation, RBAC denial, etc.
    end

    PW-->>CI: HTML report + traces on failure
```text

**Key decision 2026**: Use **Option B (real signup flow)** in at least 1–2 security-focused specs.
Use Option A in the setup project itself for speed.
````
