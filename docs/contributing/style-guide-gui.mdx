# GUI Styling & UX Guide

This document is the single source of truth for all GUI, styling, and UX patterns within SveltyCMS. Adhering to this guide is mandatory for all contributions to ensure a cohesive, accessible, and performant user interface.

## 🏗️ Foundation: Native Components + Tailwind + Theme System

Our UI is built on a robust and flexible foundation:

- **`src/components/ui/`**: 37+ reusable, WCAG 3.0-ready Svelte 5 primitives. **Always prefer these.**
- **Tailwind CSS v4** with CSS-first configuration (`@import "tailwindcss"` in `app.css`).
- **Admin Theme System**: Full support for color + structural theming inspired by Drupal Gin (see `admin-theme-plan.mdx`).

**Key Rules**:

- Use suffix `!` for important utilities (`py-0!`, `px-1!`).
- Prefer RTL logical properties (`ps-`, `pe-`, `start-`, `end-`).
- Never use hardcoded colors — always use presets or CSS variables.
- **No `tailwind.config.js`**: CSS-based config only (`@import "tailwindcss"` in `app.css`).
- **No `@apply`** except for base layer component abstractions; write utility classes inline.
- **Design Tokens**: Pure Tailwind v4 `@theme` block provides all semantic palettes (`primary`, `secondary`, `tertiary`, `surface`, `error`, `warning`, `success`). Preset classes (`preset-filled-*`, `preset-tonal-*`, `preset-outlined-*`) are defined as `@utility` classes.
- **Base Classes**: `card`, `badge`, `btn`, `input`, `btn-icon` are defined as `@utility` classes in `app.css`.

## 🎨 Core Principles

### Use Semantic Theme Colors

- **ALWAYS** use the preset classes defined in `app.css` (e.g., `preset-filled-primary-500`, `preset-tonal-success`, `preset-outlined-surface-500`).
- **ALWAYS** use CSS custom properties for one-off colors (e.g., `var(--color-primary-500)`, `var(--color-error-500)`).
- **NEVER** use hardcoded Tailwind colors (e.g., `text-red-500`, `bg-blue-500`).
- **ALWAYS** follow the **Dynamic Accent Color Pair Policy** (configurable per theme):
  - Default: **Tertiary (Blue)** in light mode and **Primary (Green)** in dark mode.
  - Implement as: `text-tertiary-500 dark:text-primary-500` / `preset-filled-tertiary-500 dark:preset-filled-primary-500`.
  - Can be overridden by admin themes via `AdminTheme` context.
  - Hardcoding `primary-500` or `tertiary-500` without a dark-mode counterpart is prohibited.

### Use Native UI Components + Theme Context

- **ALWAYS** check `src/components/ui/` first before building any UI element. The library covers buttons, inputs, modals, drawers, popovers, tabs, tables, tree-views, and more.
- **NEVER** re-implement a primitive that already exists as a native component. Extend via `$props()` and snippets instead.
- All components must respect the active `AdminTheme` context (density, variant, radii, etc.) via `getContext(ADMIN_THEME_KEY)` from `theme-context.svelte.ts`.

### Svelte 5 Runes Only

All state management must use Svelte 5 Runes (`$state`, `$derived`, `$effect`, `$props`, `$bindable`). Svelte 4 legacy syntax (`export let`, `$:`, stores) is prohibited.

### Dark Mode First

All components must include `dark:` variants for every color, background, and border class. The UI must be perfectly usable in both modes.

### WCAG 3.0 Accessibility

- Proper semantic HTML (`<fieldset>`, `<legend>`, `<dialog>`, `<nav>`)
- ARIA attributes (`aria-invalid`, `aria-describedby`, `aria-selected`, `aria-expanded`)
- `role="alert"` on error messages, `role="status"` on live regions
- Minimum 24px touch targets on interactive elements
- `prefers-reduced-motion` support on all animations
- All interactive elements must have accessible names (`aria-label` or associated `<label>`)

### Performance

Avoid expensive calculations. Use `$derived` and `$derived.by` for computed values. Use snippets (`{@render}`) for content customization instead of slot-based patterns.

## 📦 Native Component Library Quick Reference

Always import from `@components/ui/`. Key components:

| Category        | Components                                                                                                                                                                   |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Actions**     | `Button`, `Dropdown`, `Tooltip`                                                                                                                                              |
| **Forms**       | `Input`, `FloatingInput`, `Checkbox`, `RadioGroup` + `RadioItem`, `Select`, `Combobox`, `DatePicker`, `FileUpload`, `Rating`, `SegmentedControl`, `Slider`, `Tags`, `Toggle` |
| **Layout**      | `Modal`, `Drawer`, `Popover`, `Portal`, `Card`                                                                                                                               |
| **Navigation**  | `Breadcrumb`, `Stepper`, `Tabs` (compound), `TreeView`                                                                                                                       |
| **Data**        | `Table` + `Table.Filter` + `Table.Pagination` + `Table.Status`                                                                                                               |

| **Interactive** | `Accordion` + `AccordionItem`, `Collapsible`                                                                                                                                 |

### Usage Pattern

All native components follow the same Svelte 5 pattern:

````svelte
<script>
  import Button from '@components/ui/button.svelte';
  import Modal from '@components/ui/modal.svelte';
  import Input from '@components/ui/input.svelte';

  let open = $state(false);
  let name = $state('');
</script>

<Button variant="primary" onclick={() => open = true}>Open Modal</Button>

<Modal bind:open title="Edit Profile">
  <Input bind:value={name} label="Name" />
  {#snippet footer()}
    <Button variant="primary" onclick={() => open = false}>Save</Button>
  {/snippet}
</Modal>
```text

## Layout & Spacing

### Widget Containers

All form input widgets (Text, Relation, RichText, etc.) follow a standard layout.

- **Wrapper**: `relative mb-4 w-full`
- **Error Spacing**: Add `pb-6` to the wrapper if using Pattern 1 (Absolute) error messages. This provides space for error text without shifting layout.

```svelte
<div class="relative mb-4 w-full pb-6">
  <Input bind:value={field} label="Title" error={fieldError} />
</div>
```text

## ⌨️ Form Inputs

### Standard Inputs (Text, Select)

Use native `Input`, `Select`, or `FloatingInput` components. They handle labels, error states, and ARIA automatically.

```svelte
<Input bind:value={title} label="Page Title" error={errors.title} required />
<Select bind:value={status} label="Status" options={statusOptions} />
<FloatingInput bind:value={email} label="Email" type="email" icon="mdi:email" />
```text

### Complex Inputs (RichText, MediaUpload)

For complex inputs that are their own self-contained "app," use a bordered container with surface colors:

- **Container**: `relative w-full rounded border border-surface-200 bg-surface-50 shadow-sm dark:border-surface-700 dark:bg-surface-900`
- **Internal Structure**: Use `border-b border-surface-200 dark:border-surface-700` to separate toolbars and content areas.

```svelte
<div class="relative w-full rounded border border-surface-200 bg-surface-50 shadow-sm dark:border-surface-700 dark:bg-surface-900">
  <div class="toolbar flex flex-wrap border-b border-surface-200 dark:border-surface-700 p-2 gap-1">
    <!-- toolbar buttons -->
  </div>
  <div class="editor-content p-4">
    <!-- editor area -->
  </div>
</div>
```text

## Validation & Error Handling

Three distinct error patterns. Choose the correct one for your context.

### Pattern 1: Absolute Error Text (Default)

For most inputs. Error appears below without shifting layout. All native form components (`Input`, `Select`, `FloatingInput`, `Combobox`) handle this automatically via their `error` prop.

- **When to Use**: Text, Select, Combobox, DatePicker, FileUpload.
- **How**: Pass `error="message"` to the component. No manual positioning needed.

```svelte
<Input bind:value={title} label="Title" error="Title is required" />
```text

### Pattern 2: Static Error Text

For components where absolute positioning is awkward (radio groups, checkboxes).

- **When to Use**: RadioGroup, Checkbox group.
- **How**: Display error below the component group.

```svelte
<RadioGroup bind:value={plan} label="Select Plan">
  <RadioItem value="free" label="Free" />
  <RadioItem value="pro" label="Pro" />
</RadioGroup>

{#if errors.plan}
  <p class="mt-2 text-xs text-error-500" role="alert">{errors.plan}</p>
{/if}
```text

### Pattern 3: Error Banner

For complex, multi-part components where a single line is insufficient.

- **When to Use**: Mega menu builder, complex list inputs.
- **Classes**: `flex items-center gap-2 rounded border border-error-200 bg-error-50 p-3 text-error-700 dark:border-error-800 dark:bg-error-900/20 dark:text-error-300`

```svelte
{#if errors.form}
  <div class="flex items-center gap-2 rounded border border-error-200 bg-error-50 p-3 text-error-700 dark:border-error-800 dark:bg-error-900/20 dark:text-error-300" role="alert">
    <iconify-icon icon="mdi:alert-circle" width="16"></iconify-icon>
    {errors.form}
  </div>
{/if}
```text

## 🔘 Buttons & Actions

Use the native `Button` component. It handles variants, sizes, loading state, and icons automatically.

```svelte
<!-- Primary action (Save, Submit) -->
<Button variant="primary">Save</Button>

<!-- Secondary action -->
<Button variant="secondary">Cancel</Button>

<!-- Tertiary/brand action -->
<Button variant="tertiary">Add New</Button>

<!-- Ghost action (toolbar buttons) -->
<Button variant="ghost" leadingIcon="mdi:cog">Settings</Button>

<!-- Destructive action -->
<Button variant="error" leadingIcon="mdi:delete">Delete</Button>

<!-- Outline action -->
<Button variant="outline">Back</Button>

<!-- With loading state -->
<Button variant="primary" loading>Submitting...</Button>

<!-- Icon-only -->
<Button variant="ghost" leadingIcon="mdi:plus" aria-label="Add item" />
```text

**Available variants**: `primary`, `secondary`, `tertiary`, `surface`, `success`, `warning`, `error`, `ghost`, `outline`
**Available sizes**: `sm`, `md` (default), `lg`, `xl`

## 🌫️ Overlays (Modals, Drawers, Popovers)

Use native overlay components — they handle backdrop, blur, focus trapping, and accessibility automatically.

### Modal

```svelte
<Modal bind:open title="Edit Settings" size="lg">
  <!-- body content -->
  {#snippet footer()}
    <Button variant="primary" onclick={() => open = false}>Save</Button>
    <Button variant="ghost" onclick={() => open = false}>Cancel</Button>
  {/snippet}
</Modal>
```text

### Drawer

```svelte
<Drawer bind:open position="right" title="Details">
  <!-- drawer content -->
  {#snippet footer()}
    <Button variant="primary">Confirm</Button>
  {/snippet}
</Drawer>
```text

### Popover / Dropdown

```svelte
<Dropdown bind:value={selected} {options} position="bottom">
  {#snippet trigger()}
    <Button variant="outline" trailingIcon="mdi:chevron-down">
      {selectedLabel || 'Select...'}
    </Button>
  {/snippet}
</Dropdown>
```text

## 🌙 Dark Mode

Every class that sets a color, background, or border must have a `dark:` variant using theme colors. Our native components handle this automatically.

When writing custom styles:

- **Borders**: `border-surface-200 dark:border-surface-800`
- **Backgrounds**: `bg-surface-50 dark:bg-surface-950`
- **Text**: `text-surface-700 dark:text-surface-200`

## 📄 Unified Page Structural Architecture

To stop layout drift and prevent interfaces from looking different across views, every single view in SveltyCMS must match this precise structural layer blueprint exactly. No raw background overrides or variable wrapper styles are permitted.

### 1. View Root Shell

Every workspace page must be enclosed within an absolute shell mapping safely to the active admin theme containers. No raw background selections are allowed.

```svelte
<div class="absolute inset-0 p-6 space-y-6 bg-surface-50 dark:bg-surface-950 overflow-y-auto admin-theme-container">
  <!-- All page content goes here -->
</div>
```text

### 2. Standard Page Header

Every configuration workspace must invoke the global `<PageTitle>` primitive. Writing custom or inline `<h1>` variants is forbidden. Actions are dropped strictly inside the default layout slot to populate on the trailing edge.

```svelte
<script>
  import PageTitle from '@components/page-title.svelte';
</script>

<PageTitle name="Extension Management" icon="mdi:form-select" description="Install and manage plugins, widgets, and custom workspace extensions">
  <Button variant="primary" leadingIcon="mdi:plus">Install Extension</Button>
</PageTitle>
```text

**Rules**:

- Use `<PageTitle>` for every page header — never an inline `<h1 class="text-3xl …">`.
- `name` = title; `icon` = leading icon (defaults to `text-tertiary-500 dark:text-primary-500`).
- `description` = optional subtitle (renders as `text-sm font-medium opacity-50`).
- Action buttons go in the default slot — they render on the right.

### 3. Structural Segmented Containers (The Card Constraint)

All content grids, data tables, metrics feeds, and system blocks must reside inside an explicit bounding card. No content may lay bare on the application viewport background.

```svelte
<div class="card p-6 border border-surface-200 dark:border-surface-800 bg-white dark:bg-surface-900/40 backdrop-blur-md shadow-xs"
     style="border-radius: var(--admin-radius-card, 0.75rem);">
  <!-- Tables, grids, forms, stats — everything goes inside a card -->
</div>
```text

**Rules**:

- Always wrap content blocks in cards, never bare `<div>` sections.
- Use `space-y-4` or `space-y-6` inside cards for internal spacing.
- Always include `style="border-radius: var(--admin-radius-card, 0.75rem);"` to respect the active admin theme.

## 💎 Enterprise Layout Principles

Three rules for high-density admin interfaces that look professional in both light and dark mode.

### 1. Inactive Controls Should Recede ("Quiet Forms")

When an option isn't selected or a toolbar filter isn't active, blend it into the surface so the user's eye stays pinned on the workspace. Inactive controls should match the surrounding background layer until hovered or engaged.

```svelte
<!-- ✅ Good: inactive button blends into background -->
<button class="text-surface-500 hover:text-surface-900 dark:text-surface-500 dark:hover:text-white bg-transparent">
  Horizontal Toolbar
</button>

<!-- ❌ Bad: inactive state has full accent styling — competes for attention -->
<button class="text-primary-500 bg-primary-100 dark:text-primary-400 dark:bg-primary-500/20">
  Horizontal Toolbar
</button>
```text

### 2. Border-First Separation

Use slim, high-contrast borders to separate content regions instead of heavy background fills. This keeps layouts crisp at high data density and anchors user movement across multi-column configurations.

```svelte
<!-- ✅ Good: border defines the boundary, background stays light -->
<div class="rounded border border-surface-200 p-4 bg-surface-50/30 dark:border-surface-800 dark:bg-surface-900/20">
  <!-- grouped content -->
</div>

<!-- ❌ Bad: relies on heavy background tint for separation, no border anchor -->
<div class="rounded bg-surface-100 p-4 dark:bg-surface-800">
  <!-- grouped content -->
</div>
```text

### 3. Adaptive Accent Saturation

In light mode, use deeply anchored, rich accent tones for crisp visibility. In dark mode, reduce saturation and lift lightness — full-saturation accents that look premium on a light canvas become harsh and cause eye strain against dark backgrounds.

```svelte
<!-- ✅ Good: light mode uses rich tertiary tone, dark mode uses subdued primary -->
<button class="preset-filled-tertiary-500 dark:preset-filled-primary-500">
  Save Changes
</button>

<!-- ❌ Bad: same full-saturation accent in both modes -->
<button class="preset-filled-primary-500">
  Save Changes
</button>
```text

> The preset system already enforces per-mode saturation control. For custom accent utilities, target `oklch` lightness ~55–65% in light mode; step saturation down and lift lightness 5–10 points in dark mode. Never hardcode raw `oklch` values directly in components.

## 📐 Data Presentation Patterns

### Grid Dashboards & Configurations

When presenting menu options or macro parameters, keep items locked into explicit content templates. Avoid vast sizing discrepancies by mapping actions to standard grid columns.

```svelte
<div class="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-4">
  <a href="/config/settings" class="card p-5 hover:border-tertiary-500 dark:hover:border-primary-500 transition-all flex items-start gap-4">
    <iconify-icon icon="mdi:cog" width="24" class="text-surface-500"></iconify-icon>
    <div>
      <h3 class="text-sm font-semibold text-surface-900 dark:text-white">System Settings</h3>
      <p class="text-xs text-surface-500 mt-1">Configure global platform attributes</p>
    </div>
  </a>
  <!-- Repeat for each grid item -->
</div>
```text

### Data Management Tables

Tables must scale natively across view bounds and feature explicit alignment breaks. Rows must feature subtle divider lines rather than bold grid matrices. Always wrap tables in `overflow-x-auto`.

```svelte
<div class="overflow-x-auto w-full">
  <table class="w-full text-sm border-collapse">
    <thead>
      <tr class="border-b border-surface-200 dark:border-surface-800 text-left text-xs uppercase tracking-wider text-surface-400">
        <th class="pb-3 font-semibold">Service Name</th>
        <th class="pb-3 font-semibold">Status</th>
        <th class="pb-3 font-semibold text-right">Performance</th>
      </tr>
    </thead>
    <tbody class="divide-y divide-surface-100 dark:divide-surface-800/60">
      <tr class="text-surface-700 dark:text-surface-200 hover:bg-surface-50/40 dark:hover:bg-surface-900/30">
        <td class="py-3 font-medium">database_service</td>
        <td class="py-3"><span class="badge preset-tonal-primary-500">Active</span></td>
        <td class="py-3 text-right font-mono text-xs text-surface-400">12ms</td>
      </tr>
    </tbody>
  </table>
</div>
```text

### Status Badges

```svelte
<span class="badge text-[10px] preset-filled-success-500">Active</span>
<span class="badge text-[10px] preset-tonal-primary-500">In Progress</span>
```text

### Quick Action Links

```svelte
<a href="/config/target" class="btn btn-sm preset-tonal-surface-500 w-full justify-between">
  <span>Label</span>
  <iconify-icon icon="mdi:arrow-right"></iconify-icon>
</a>
```text

## 🛑 Complete Core Anti-Patterns (DO NOT USE)

- ❌ `container mx-auto p-4` — use `absolute inset-0 p-6` at the layout container entry point
- ❌ Inline `<h1 class="text-3xl font-bold …">` page header — use the `<PageTitle>` component; no custom inline title variants permitted
- ❌ Bare `<div>` content sections placed directly on the viewport background — always wrap in a `.card`
- ❌ `table-container` / `table-hover` classes — use the native `overflow-x-auto` + `divide-y` pattern
- ❌ `$app/stores` (`$page`) — use `$app/state` (`page`)
- ❌ `variant-ghost-surface` / `variant-filled-primary` — use `preset-*` classes or native `<Button>`
- ❌ Raw color assignments (`bg-black`, `bg-neutral-900`, `text-red-500`) — use preset classes or CSS custom properties exclusively
- ❌ Full-saturation accent buttons in dark mode without a `dark:` variant — combine `preset-filled-tertiary-500 dark:preset-filled-primary-500`
- ❌ Heavy background fills (`bg-surface-100 dark:bg-surface-800`) for container separation — use `border border-surface-200 dark:border-surface-800` with translucent backgrounds
- ❌ Active accent styling on unselected/inactive segment controls — use `text-surface-500 bg-transparent` for dormant options
- ❌ Unbounded table regions placed directly on the page without `overflow-x-auto` wrapper

## ✅ Style Checklist

Before submitting a PR:

- [ ] **Theme Awareness**: Component respects `--admin-*` CSS variables and `AdminTheme` context (density, variant, radii). CSS variable fallbacks included (e.g., `var(--admin-radius-card, 0.75rem)`).
- [ ] **Use Native Components**: Prefer `src/components/ui/` over building from scratch.
- [ ] **Semantic Colors**: Use preset classes (`preset-filled-primary-500`) or CSS variables (`var(--color-primary-500)`). No hardcoded hex/Tailwind colors.
- [ ] **Dynamic Accent Pairing**: Interactive accents use `tertiary-500` (light) / `primary-500` (dark), or are intentionally overridden by AdminTheme.
- [ ] **Dark Mode**: All custom styles have `dark:` variants.
- [ ] **Error Handling**: Correct error pattern used (Absolute via component `error` prop, Static, or Banner).
- [ ] **WCAG 3.0**: `aria-label` on icon-only buttons, `role="alert"` on errors, `role="status"` on live regions, 24px touch targets.
- [ ] **`prefers-reduced-motion`**: Any custom animation checks the media query.
- [ ] **Svelte 5 Runes**: `$state`, `$derived`, `$props`, `$bindable` used. No Svelte 4 legacy syntax.
- [ ] **Responsiveness**: Mobile-first, tested at 320px width.
- [ ] **Tailwind v4 & RTL Conventions**: Use suffix exclamation marks (e.g., `py-0!`) and RTL logical properties (e.g., `ps-`, `pe-`).
````
